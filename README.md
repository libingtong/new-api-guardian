# New API Guardian

`New API Guardian` 是一个面向 [new-api](https://github.com/QuantumNous/new-api) 的外挂式运营分析与稳定性治理控制台。

它解决的不是“看日志”这么简单，而是两类在生产环境里都会长期出现的问题：

第一类是“看不清”：

- 哪些 Token 在消耗最多资源
- 哪些用户最活跃
- 哪些模型使用最集中
- 最近请求和错误到底发生在什么地方
- 某个 IP、某个 Token、某个模型的行为是否异常

第二类是“出了问题没人及时处理”：

- 某个渠道开始连续报错，但你没有第一时间发现
- 某个模型已经坏了，却还挂在渠道上持续被选中
- 手工禁用渠道太慢，恢复又全靠人工记忆
- `new-api` 开了内存缓存，数据库改了但运行态没有马上生效
- 很多错误其实只是调用方式不匹配，例如“只支持流式”的上游被非流式请求打坏，结果整条渠道被误伤

这个项目的目标，就是把这些日常运维痛点自动化、可视化、可审计化。

## 它能做什么

### 0. 它不只是熔断工具

这个项目不是一个单纯的“自动禁用渠道脚本”。

它实际上把两套能力放在了一起：

- 运营分析
- 稳定性治理

也就是说，你可以把它理解成：

> `new-api` 的分析控制台 + 熔断恢复控制台。

很多人会先因为“排行榜、明细、模型分布、IP 分析”安装它，后面再逐步启用规则治理能力。

### 1. 运营分析与可视化
- 使用量总览：请求数、Quota、Tokens、成功率、错误率、平均耗时
- Token / 用户 / 模型排行榜
- 独立排行榜页，适合做运营看板
- 最近请求明细，便于快速定位异常
- 模型分布分析
- IP 维度统计，辅助排查异常来源
- 支持按时间范围、Token、用户、模型、分组筛选

### 2. 读日志，但不止于读日志
- 直接连接 `new-api` 的 MySQL
- 增量扫描 `logs.type = 5` 的错误日志
- 根据规则判断是否需要治理动作
- 所有命中、禁用、恢复都有审计记录

### 3. 自动熔断渠道
- 命中规则后不会立刻误伤
- 会先主动调用 `new-api` 探测接口做一次二次确认
- 只有主动探测也失败，才会真正禁用渠道
- 禁用后自动进入恢复探测队列

### 4. 自动下线单个模型，而不是一刀切整条渠道
- 支持 `disable_model`
- 对模型级问题，只移除该模型，不必直接禁整条渠道
- 同步维护：
  - `channels.models`
  - `abilities`
- 避免“页面上看起来删掉了，选路却还在命中”的假状态

### 5. 自动恢复
- 渠道级恢复：每 10 秒探测一次，连续 3 次成功后恢复
- 模型级恢复：每 10 秒探测一次被移除模型，连续 3 次成功后写回
- 如果某个渠道是因为“模型被删空”才被禁用，只要任意一个模型恢复成功，渠道就会一起恢复

### 6. 刷新 `new-api` 运行态缓存
- 不只是改数据库
- 在禁用/恢复渠道或模型后，可选调用 `new-api` 官方接口刷新渠道缓存
- 避免开启 `MEMORY_CACHE_ENABLED` 时，数据库已经变了但运行态还没同步

### 7. 一个真正可用的管理台
- 规则列表
- 新增/编辑/删除规则
- 渠道级恢复状态
- 模型级恢复状态
- 动作审计
- 规则命中记录
- 后台隐藏路径
- IP 白名单
- 管理页密码验证

### 8. 企业微信群机器人告警
- 当某个渠道被判定为“不稳定渠道”并进入人工恢复模式时，自动向企业微信群机器人发送告警
- 告警会带上：
  - 渠道名称 / 渠道 ID
  - 触发规则
  - 统计窗口和阈值
  - 最近失败模型
  - 错误码 / 状态码
  - 最近错误内容
  - 关联日志 ID
- webhook 发送失败不会阻断禁用动作，避免告警链路反过来影响治理主流程

## 为什么它适合 `new-api`

这个项目不是去改 `new-api` 本体，而是做一个“外挂式控制层”。

这意味着：

- 你不需要 fork `new-api` 去塞一堆业务逻辑
- 规则、审计、恢复队列都由这个项目自己维护
- 它只在必要时写回 `new-api` 数据库
- 同时还能调用 `new-api` 官方接口刷新缓存

这比单纯“直接改数据库”更稳，也比“完全只靠官方 API”更灵活。

## 为什么它比官方自带自动禁用更靠谱

`new-api` 自带的自动禁用能力当然有价值，但它更偏“基础保护”，而这个项目解决的是更复杂的生产治理问题。

两者最核心的差别在这里：

### 1. 它不是看到错误就立刻禁用，而是先主动复核

这个项目的治理链路是：

1. 先读错误日志
2. 命中规则
3. 再主动调用探测接口验证一次
4. 只有探测也失败，才真正禁用

这比“单次错误直接触发禁用”更稳，特别适合这些场景：

- 上游短暂抖动
- 客户端请求参数错误
- 非流式误调用了只支持流式的模型
- 日志里报错了，但渠道本身其实还活着

### 2. 它支持模型级治理，而不只是整条渠道开关

官方自动禁用更接近“渠道坏了就关掉”。

但线上真实情况经常是：

- 渠道没坏
- 只是其中某一个模型坏了

这时最合理的动作不是禁整条渠道，而是：

- 只下线这个模型
- 保留同渠道其它还能用的模型继续服务

这正是这个项目的 `disable_model` 能力。

### 3. 它能把“误伤”和“真实故障”区分得更细

这个项目不是只看一个状态码，也不是只看一条错误消息。

规则可以组合：

- 渠道
- 分组
- 模型
- 错误文本
- 错误码
- 状态码
- 请求路径
- 窗口时间
- 命中阈值

所以它更像一套“可编排治理规则”，而不是一个固定条件触发器。

### 4. 它的恢复逻辑是内建的、连续验证的

不是禁掉就结束了。

这个项目会：

- 对禁用渠道定时探测
- 对被移除模型定时探测
- 连续成功达到阈值后自动恢复

也就是说，它不是单向熔断，而是完整的“熔断 + 恢复”闭环。

### 5. 它对 `new-api` 的缓存问题有补偿能力

很多人在线上遇到的不是“数据库没改成功”，而是：

- 数据库改了
- 但 `new-api` 开了内存缓存
- 运行态没有马上生效

这个项目支持在治理动作后，主动调用 `new-api` 官方接口刷新渠道缓存。

所以它不仅会“做决定”，还会尽量确保“决定马上在运行态生效”。

### 6. 它是可审计、可追溯的

官方自动禁用通常更像黑盒保护。

而这个项目会把整条链路都记录下来：

- 哪条规则命中了
- 哪条日志触发了
- 为什么禁用
- 为什么恢复
- 渠道级还是模型级动作

这对生产运维特别重要，因为你不只是想“让它自动跑”，你还得知道“它为什么这么跑”。

### 7. 它更适合高模型密度、高渠道密度的生产环境

当你只有少量渠道时，简单自动禁用就够用了。

但当你进入下面这种场景：

- 渠道很多
- 模型很多
- 上游行为不一致
- 某些模型只支持特定调用方式
- 你需要尽可能减少误伤

那就需要更精细的外挂治理层，而不是只靠一个统一的自动禁用开关。

一句话总结：

> 官方自动禁用更像“保险丝”，而这个项目更像“带规则引擎、主动探测、模型级治理、自动恢复和审计能力的生产级熔断系统”。

## 核心思路

整个系统只有两条主线：

1. 错误治理
- 扫描错误日志
- 命中规则
- 主动探测二次确认
- 执行 `disable_channel` 或 `disable_model`

2. 自动恢复
- 对已禁用渠道做恢复探测
- 对已移除模型做恢复探测
- 连续成功达到阈值后自动恢复

一句话概括：

> 先看日志，再主动验证，最后才真正执行治理动作。

## 解决的典型误伤场景

这是这个项目一个非常重要的设计点。

很多线上 `500` 并不等于“渠道真的坏了”，而可能只是：

- 客户端把“只支持流式”的模型用非流式方式调用
- 某些模型只能走特定 endpoint
- 某些错误只是临时抖动，不值得马上禁用

所以现在的实现里：

- 禁用前会先主动探测
- 探测默认走 `stream=true`
- 恢复探测也默认走 `stream=true`

这能显著减少“因为调用方式不匹配而被误禁用”的情况。

## 为什么它不只是一个“自动禁用外挂”

如果只看后半部分能力，这个项目很容易被理解成：

- 读取错误日志
- 自动禁用渠道

但这会低估它的价值。

它真正适合的定位是：

- 白天看分析数据
- 晚上做稳定性治理
- 出问题时能快速排查
- 需要时又能自动执行治理动作

你可以先把它当成：

- `new-api` 的使用量分析看板
- `new-api` 的运营排行榜后台
- `new-api` 的请求明细查询台

然后再逐步开启：

- 规则治理
- 自动禁用
- 自动恢复
- 模型级下线

所以它不是一个“只能在故障时才有价值”的工具，而是一个可以日常持续使用的后台系统。

```bash
cp .env.example .env
```

2. 修改数据库连接和后台安全配置

3. 启动

```bash
docker compose up -d --build

如果你希望在“不稳定渠道”被禁用时自动提醒人工处理，可以额外配置企业微信群机器人：

- 管理页：`http://localhost:8000${ADMIN_PATH}`

## Docker 推荐配置

下面是最常见的接入方式：

```yaml
services:
  analytics-ui:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8000:8000"
    environment:
      - DB_HOST=host.docker.internal
      - DB_PORT=3306
      - DB_USER=oneapi
      - DB_PASSWORD=your_password
      - DB_NAME=oneapi
      - APP_HOST=0.0.0.0
      - APP_PORT=8000
      - ADMIN_PATH=/console
      - ALLOWED_IPS=127.0.0.1,192.168.1.0/24
      - TRUST_PROXY_HEADERS=1
      - ADMIN_PASSWORD=change_me
      - INIT_IMPORT_FROM_DUMP=0
      - CHANNEL_TEST_STREAM=1
      - NEW_API_BASE_URL=http://host.docker.internal:3000
```

说明：

- `NEW_API_BASE_URL` 配置后，系统会在治理动作后尝试刷新 `new-api` 渠道缓存
- 如果没有显式配置 `NEW_API_ACCESS_TOKEN` 和 `NEW_API_USER_ID`
- 系统会自动从 `users` 表读取一个已启用管理员的 `access_token + id`

### 这些 Docker 参数为什么这样配

下面这组参数不是“随便填的样板”，而是针对最常见的部署形态设计的：

- `new-api` 运行在宿主机或另一套容器里
- 本项目作为外挂服务单独启动
- 需要连同一个 MySQL
- 需要在治理动作后顺手刷新 `new-api` 缓存

#### 数据库连接

- `DB_HOST=host.docker.internal`
  - 含义：容器从宿主机网络访问 MySQL
  - 为什么这样配：很多人的 `new-api` 数据库就在宿主机或另一套独立服务里，不在当前 compose 网络内
  - 什么时候要改：
    - 如果 MySQL 和本服务在同一个 compose 网络里，改成 MySQL 服务名，例如 `mysql`
    - 如果数据库是云数据库，改成实际数据库地址

- `DB_PORT=3306`
  - 含义：MySQL 端口
  - 为什么这样配：MySQL 默认就是 `3306`
  - 什么时候要改：
    - 如果你映射成了别的端口，例如 `13306`，就改成对应端口

- `DB_USER=oneapi`
  - 含义：连接 `new-api` 数据库的账号
  - 为什么这样配：示例里假设你给 `oneapi` 库单独建了一个业务账号
  - 什么时候要改：
    - 如果你使用的是 `root`
    - 或者你的数据库账号名不是 `oneapi`

- `DB_PASSWORD=your_password`
  - 含义：上面这个数据库账号的密码
  - 为什么这样配：这里只是占位值，必须换成真实密码

- `DB_NAME=oneapi`
  - 含义：`new-api` 使用的数据库名
  - 为什么这样配：大多数部署都直接叫 `oneapi`
  - 什么时候要改：
    - 如果你的库名不是 `oneapi`

#### Web 服务监听

- `APP_HOST=0.0.0.0`
  - 含义：容器内监听所有网卡
  - 为什么这样配：Docker 容器如果只监听 `127.0.0.1`，外部端口映射访问不到
  - 基本不建议改

- `APP_PORT=8000`
  - 含义：容器内 Web 服务端口
  - 为什么这样配：当前镜像和 compose 都按 `8000` 组织
  - 什么时候要改：
    - 只有当你明确希望容器内服务监听别的端口时才需要改

#### 管理页安全

- `ADMIN_PATH=/console`
  - 含义：管理页隐藏路径
  - 为什么这样配：避免后台入口直接暴露在显眼路径上
  - 建议：
    - 生产环境不要用太常见的路径
    - 可以改成更长、更不容易猜的路径

- `ALLOWED_IPS=127.0.0.1,192.168.1.0/24`
  - 含义：允许访问页面和后台接口的 IP / CIDR 白名单
  - 为什么这样配：后台属于高权限界面，不应该对所有来源开放
  - 示例含义：
    - `127.0.0.1` 允许本机访问
    - `192.168.1.0/24` 允许同一内网网段访问
  - 什么时候要改：
    - 如果你通过公网反代访问，就填你的办公出口 IP
    - 如果你前面挂了 Nginx，建议只允许反代后的真实来源 IP
  - 留空表示不限制，但不建议在公网环境这么做

- `TRUST_PROXY_HEADERS=1`
  - 含义：信任 `X-Forwarded-For` / `X-Real-IP`
  - 为什么这样配：很多线上部署前面都有 Nginx 或其他反代，真实用户 IP 在请求头里
  - 什么时候应该开：
    - 你的请求确实是从可信反代转发过来的
  - 什么时候要小心：
    - 如果应用端口直接裸露到公网，攻击者可以伪造这些头，白名单判断会不可靠

- `ADMIN_PASSWORD=change_me`
  - 含义：管理页登录密码
  - 为什么这样配：后台除了隐藏路径和 IP 白名单，还应该再加一层密码保护
  - 必须改成强密码
  - 如果留空：
    - 不启用后台密码验证
    - 这时就只剩隐藏路径和 IP 白名单

#### 初始化策略

- `INIT_IMPORT_FROM_DUMP=0`
  - 含义：是否允许容器启动时从 SQL dump 导入主业务表
  - 为什么这样配：这个项目是外挂服务，不应该接管 `new-api` 主库初始化
  - 推荐保持 `0`
  - 只有在你明确想让它补主业务表时才考虑改成 `1`

#### 探测策略

- `CHANNEL_TEST_STREAM=1`
  - 含义：调用 `new-api` 探测接口时默认带 `stream=true`
  - 为什么这样配：很多模型或某些上游只支持流式；如果探测默认非流式，很容易把“调用方式不匹配”误判成“渠道坏了”
  - 推荐保持 `1`
  - 只有当你明确确认你的探测目标必须非流式时才考虑改成 `0`

#### `new-api` 缓存刷新

- `NEW_API_BASE_URL=http://host.docker.internal:3000`
  - 含义：外挂服务调用 `new-api` 管理接口的地址
  - 为什么这样配：示例假设 `new-api` 服务运行在宿主机 `3000` 端口
  - 作用：
    - 在禁用/恢复渠道或模型后调用 `PUT /api/channel/`
    - 让 `new-api` 立即重建渠道缓存
  - 什么时候要改：
    - 如果 `new-api` 在同一个 compose 网络内，改成服务名，例如 `http://new-api:3000`
    - 如果 `new-api` 通过其他域名访问，就填实际地址

#### 没写出来但你可能会关心的两个参数

- `NEW_API_ACCESS_TOKEN`
  - 含义：显式指定用于刷新缓存的管理令牌
  - 如果不填：
    - 系统会自动从 `users` 表里挑一个已启用管理员的 `access_token`

- `NEW_API_USER_ID`
  - 含义：和上面的访问令牌配套使用的管理员用户 ID
  - 如果不填：
    - 系统会与数据库里自动解析到的管理员令牌一起回退获取

### 企业微信群机器人

- `WECOM_ROBOT_WEBHOOK`
  企业微信群机器人的 webhook 地址。留空则不发送群通知。
- `WECOM_NOTIFY_TIMEOUT_SECONDS`
  告警请求超时时间，默认 `5` 秒。

触发时机：

- 仅在渠道被判定为 `disable_unstable_channel`，并且数据库事务已经成功提交后发送。
- 普通的 `disable_channel`、`disable_model`、自动恢复不会发送这类人工处理告警。

设计取舍：

- 告警是附加能力，不是主链路。
- 即使企业微信群 webhook 超时、报错或返回失败，也不会回滚禁用动作。

## `new-api` 缓存刷新机制

当配置了 `NEW_API_BASE_URL` 后，本项目会在以下动作提交数据库后，调用：

```text
PUT /api/channel/
```

用于触发 `new-api` 重建渠道缓存：

- 自动禁用渠道
- 自动恢复渠道
- 自动下线模型
- 自动恢复模型

鉴权策略：

1. 优先使用显式配置
- `NEW_API_ACCESS_TOKEN`
- `NEW_API_USER_ID`

2. 探测接口和缓存刷新接口共用同一套身份来源
- 禁用前主动探测
- 自动恢复探测
- `new-api` 渠道缓存刷新

3. 如果未配置，则自动从数据库回退读取
- `users.access_token`
- `users.id`

4. 如果找不到可用管理员访问令牌
- 仍会继续写数据库
- 但不会执行探测调用和缓存刷新
- 管理页会给出显式告警

## 规则能力

每条规则都可以独立配置：

- 作用渠道
- 作用分组
- 作用模型
- 错误文本关键字
- 错误码
- 状态码
- 请求路径
- 窗口秒数
- 命中阈值
- 动作类型
- 启用/停用
- 优先级

系统禁止保存“全空条件规则”，避免一条规则误伤整库。

## 适用人群

如果你符合下面任意一种情况，这个项目就很适合你：

- 你在生产环境里使用 `new-api`
- 你挂了很多渠道和很多模型
- 你不想手动盯错误日志
- 你需要自动禁用、自动恢复、自动审计
- 你想把“模型坏了”与“渠道整体坏了”分开处理

## 边界与风险

这个项目不是只读分析器，而是会真实写回运行态配置。

上线前请明确：

- 它会修改 `channels` 和 `abilities`
- 它会真实影响 `new-api` 选路
- 规则配置过宽仍然有误伤风险

建议上线方式：

1. 先接入只读观察
- 先启动服务，先不开宽规则

2. 先启用一条非常窄的规则
- 限定渠道
- 限定模型
- 限定错误文本或状态码

3. 观察审计和恢复链路
- 确认行为符合预期后再逐步放开

## 项目定位

这个项目适合被理解为：

> `new-api` 的运营分析层、错误治理层、自动熔断层、自动恢复层、运维可视化层。

它不是替代 `new-api`，而是补上 `new-api` 在复杂生产环境里的一个关键缺口。
